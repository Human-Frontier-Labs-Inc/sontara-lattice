# Playbook: broker-server Total Compromise

**Severity:** Catastrophic
**Scope:** Entire fleet -- total loss of trust
**Last updated:** 2026-03-28

## Scenario

broker-server (<broker-ip>, Ubuntu 24.04, 32GB RAM) is fully compromised. An attacker has root or user-level shell access and controls the machine.

This is the game-over scenario for the current architecture. broker-server is the single point of failure for the entire Sontara Lattice. It runs every critical service. There is no redundancy and no out-of-band verification path that the attacker cannot also compromise.

## What the Attacker Controls

| Service | What It Does | What the Attacker Gets |
|---|---|---|
| **Broker** (port 7899) | All peer registration, messaging, events, fleet memory, machine health | Can see every active Claude session, read/send all messages, modify fleet memory, manipulate health scores, unquarantine any machine, suppress quarantine events |
| **NATS** (port 4222) | All fleet event streaming via JetStream | Can publish fake events, subscribe to all events, suppress real events, inject poisoned data into every subscriber |
| **Wazuh Manager** | Security alert aggregation from all agents | Can suppress alerts, modify rules, disable detection, see all security events across the fleet |
| **Supervisor** | Runs all 6 LLM daemons | Can modify daemon prompts, inject malicious agent files, run arbitrary code as daemon processes |
| **Response Daemon** | Automated incident response | Can suppress incident responses, prevent email alerts, cancel IP blocks, unquarantine compromised machines |
| **Security Watch** | Event correlation and escalation | Can suppress correlation rules, prevent distributed attack detection, disable email alerting |
| **Dream Watch** | Fleet memory generation | Can poison fleet memory with arbitrary instructions that get synced to all machines |
| **Wazuh Bridge** | Wazuh-to-NATS event bridge | Can filter which Wazuh alerts reach NATS, effectively blinding the security pipeline |
| **LiteLLM Proxy** (port 4000) | Routes LLM calls to Anthropic/Vertex AI | Has all API keys. Can intercept/modify LLM requests from daemons, exfiltrate prompts and responses |
| **Syncthing** | Real-time file sync to workstation | Can modify synced files (projects, hfl-projects, ricing-resources) and the changes propagate to workstation automatically |
| **SSH keys** | Access to all fleet machines | Can SSH to workstation, edge-node, workstation-2, iot-device, laptop-1 |
| **Cloudflare tunnel** | Exposes services to your-domain.example.com | Can serve malicious content to anyone accessing the test domain, including HFL clients |

## Why This Is Catastrophic

### The Attacker Controls the Detection System

The security pipeline is: Wazuh alerts -> Wazuh Bridge -> NATS -> Security Watch -> Response Daemon -> Email.

Every component in this chain runs on broker-server. The attacker can:

1. Disable Wazuh agent reporting from broker-server itself
2. Modify Wazuh rules to ignore their own activity
3. Stop the wazuh-bridge so alerts never reach NATS
4. Publish fake "all clear" events to NATS
5. Modify security-watch correlation rules to suppress escalation
6. Stop the response-daemon so no emails are sent
7. Modify the email throttle to prevent further alerts
8. Unquarantine any machine via the broker API
9. Reset health scores to zero

**You cannot trust any security signal that originates from or passes through broker-server.**

### The Attacker Controls Fleet Memory

Fleet memory is generated by dream-watch, stored in the broker's SQLite database, and synced to every machine on session startup. The attacker can:

1. Write arbitrary instructions into fleet memory
2. These instructions get synced to `~/.claude/projects/-home-user/memory/fleet-activity.md` on every machine
3. Every new Claude session reads this as context
4. The attacker can instruct all Claude sessions to execute commands, install backdoors, or exfiltrate data

### The Attacker Can Modify Synced Files

Syncthing syncs `~/projects/`, `~/hfl-projects/`, and `~/ricing-resources/` between broker-server and workstation in real-time. The attacker can:

1. Modify source code in `~/projects/claude-peers/` -- changes sync to workstation
2. If you rebuild the binary from the modified source, the compromised code deploys to all machines
3. Modify HFL client code -- changes sync to workstation where you might deploy them
4. This is a stealthy supply chain attack via file sync

### The Attacker Has All the API Keys

LiteLLM proxy on port 4000 has environment variables containing:
- Anthropic API key
- Vertex AI credentials
- Any other LLM provider keys

These keys give the attacker:
- Unlimited API usage billed to your accounts
- Ability to make API calls that appear to come from your infrastructure
- Access to any model you have access to

## Detection

### The Hard Truth

If broker-server is fully compromised, there is no reliable automated detection path within the fleet. Every monitoring system routes through broker-server.

### Out-of-Band Detection Signals

These are the ONLY signals you can trust:

| Signal | How to Check | Limitations |
|---|---|---|
| **Tailscale admin panel** | https://login.tailscale.com/admin/machines | External to the fleet. Shows machine status, IPs, last-seen times. The attacker cannot manipulate this (it's Tailscale's infrastructure). |
| **Physical access to another machine** | Walk to workstation or edge-node, check manually | Not remote. Requires physical presence. |
| **Email not arriving** | If security alerts stop coming, that's itself a signal | Absence of evidence is not evidence of absence. You might just have a quiet period. |
| **Anthropic/GCP billing dashboard** | Check for unexpected API usage spikes | External to the fleet. Attacker cannot suppress these. |
| **GitHub audit log** | Check for unexpected commits, PR activity | External to the fleet. |
| **Cloudflare dashboard** | Check for unexpected tunnel activity or content changes | External to the fleet. |
| **Syncthing web UI on workstation** | Check for unexpected file changes synced from broker-server | The attacker's file modifications will show up as recent syncs. |

### Canary Signals

Things that should make you suspicious even without a direct alert:

- Gridwatch dashboard on edge-node shows all green but you just saw a security alert that should have degraded a machine
- Claude sessions report fleet memory content you didn't write
- Broker events show activity gaps (the attacker stopped the bridge/watchers)
- `claude-peers status` from workstation shows the broker is healthy but NATS events seem missing
- Syncthing on workstation shows recent changes to files you didn't touch

### What the iot-device Sentinel Cannot Do

iot-device publishes health data directly to NATS. But NATS runs on broker-server. If the attacker controls NATS, they can:
- See iot-device's sentinel data
- Drop or modify the messages
- Publish fake sentinel data that shows everything is fine

The sentinel provides no detection value when the NATS server itself is compromised.

### What Gridwatch on edge-node Cannot Do

Gridwatch polls the broker's `/machine-health` API. The broker runs on broker-server. The attacker controls the broker. They can:
- Return fake health data showing all machines as healthy
- Fake the service monitor responses
- Make Gridwatch display whatever they want

Gridwatch provides no detection value when the broker is compromised.

## Response

### Phase 1: Isolate (Do immediately)

1. **Take broker-server offline:**
   ```
   If you have physical access: unplug the network cable and power it down.
   If remote only: remove it from Tailscale via the admin panel.
   ```
   Prefer physical disconnection. Removing from Tailscale cuts mesh access but the machine could still reach the internet via its direct connection.

2. **Stop trusting ANY data that came from broker-server:**
   - Fleet memory on all machines: tainted
   - Recent broker events: tainted
   - Health scores: tainted
   - NATS events: tainted
   - Security alerts (or lack thereof): tainted

3. **Suspend Syncthing on workstation immediately:**
   ```bash
   # On workstation
   systemctl --user stop syncthing
   ```
   This prevents any further file sync from the compromised machine.

4. **Check for Syncthing damage:**
   ```bash
   # On workstation, check recent file changes in synced directories
   find ~/projects/ -mmin -120 -type f | head -50
   find ~/hfl-projects/ -mmin -120 -type f | head -50

   # Git status on key repos
   cd ~/projects/claude-peers && git status && git diff
   ```

### Phase 2: Assess from a Trusted Machine

Use workstation (or any machine that is NOT broker-server) for all investigation.

1. **Check SSH logs on other machines for broker-server connections:**
   ```bash
   # On workstation
   journalctl -u sshd --since "24 hours ago" | grep "<broker-ip>"

   # On edge-node
   ssh edge-node "journalctl -u sshd --since '24 hours ago' | grep '<broker-ip>'"
   ```

2. **Check authorized_keys on every machine:**
   ```bash
   for machine in workstation edge-node workstation-2 iot-device laptop-1; do
     echo "=== $machine ==="
     ssh $machine "cat ~/.ssh/authorized_keys" 2>/dev/null | wc -l
     ssh $machine "cat ~/.ssh/authorized_keys" 2>/dev/null
   done
   ```

3. **Check for persistence on other machines:**
   ```bash
   for machine in workstation edge-node workstation-2 iot-device; do
     echo "=== $machine ==="
     ssh $machine "crontab -l 2>/dev/null"
     ssh $machine "ls ~/.config/systemd/user/ 2>/dev/null"
     ssh $machine "find /tmp -mmin -120 -type f 2>/dev/null | head -10"
   done
   ```

4. **Check GitHub for unauthorized activity:**
   ```bash
   gh auth status
   gh api user/repos --jq '.[].full_name' | while read repo; do
     echo "=== $repo ==="
     gh api repos/$repo/events --jq '.[0:3][] | "\(.type) by \(.actor.login) at \(.created_at)"'
   done
   ```

5. **Check GCP/Vertex AI for unauthorized usage:**
   ```bash
   gcloud logging read "resource.type=aiplatform.googleapis.com" --limit 20 --format json
   ```

### Phase 3: Rebuild broker-server

This machine cannot be cleaned. You must rebuild from scratch.

1. **Reinstall the OS:** Fresh Ubuntu 24.04 install. Do not restore from backup -- the backup may be compromised.

2. **Generate all new credentials:**
   - New SSH keypair for broker-server
   - New claude-peers broker keypair and root token
   - New NATS configuration with new auth token
   - New Wazuh manager keypair and agent keys
   - New LiteLLM proxy configuration with new API keys
   - New Cloudflare tunnel token

3. **Reinstall services from known-good sources:**
   - Build claude-peers binary fresh from the git repo (verify the repo wasn't modified via git log/diff)
   - Install NATS from official releases
   - Install Wazuh from official packages
   - Configure LiteLLM from scratch

4. **Re-key every fleet machine:**
   ```bash
   for machine in workstation edge-node workstation-2 iot-device laptop-1; do
     echo "=== Re-keying $machine ==="
     # Remove old broker-server SSH key from authorized_keys
     # Generate new claude-peers credentials
     # Issue new UCAN token
     # Update NATS token in config
   done
   ```

5. **Re-enroll Wazuh agents on every machine.**

6. **Re-establish Syncthing trust:**
   - New device ID for broker-server
   - Re-pair with workstation
   - Verify shared folder contents match

### Phase 4: Rotate Everything External

1. **Anthropic API key:** Regenerate via the Anthropic dashboard
2. **Vertex AI credentials:** Rotate service account keys in GCP
3. **GitHub tokens:** Revoke all personal access tokens, re-authenticate gh CLI on all machines
4. **1Password:** Change vault passwords if they were stored on broker-server
5. **Cloudflare:** Regenerate tunnel token, review DNS records
6. **Tailscale:** Remove old broker-server device, re-add the rebuilt machine with new auth

## Prevention

### The Architectural Fix: Distribute Trust

The root cause is concentration. Every critical service runs on one machine. The fix is distribution:

| Service | Current Location | Should Be |
|---|---|---|
| Broker | broker-server | broker-server (keep, but reduce scope) |
| NATS | broker-server | Separate machine or managed NATS service (e.g., Synadia Cloud) |
| Wazuh Manager | broker-server | Separate machine. The security monitoring system MUST NOT run on the same host as the services it monitors. |
| Supervisor + Daemons | broker-server | Can stay, but with reduced privileges and sandboxing |
| Response Daemon | broker-server | Should run on a different machine from the detection system. If the detection machine is compromised, the response machine can still act. |
| Fleet Memory (authoritative copy) | broker-server (broker SQLite) | Should have a signed copy on a second machine. Detect tampering by comparing. |

### Must-Do

1. **Out-of-band heartbeat:** A simple cron job on broker-server that pings an EXTERNAL service (e.g., a UptimeRobot or a simple webhook) every 5 minutes. If the ping stops, you get notified via an external channel. The attacker would have to keep the heartbeat alive, which limits what they can do stealthily.

2. **External log shipping:** Ship Wazuh alerts to an external service (Grafana Cloud, Datadog, or even a simple syslog to another machine) in addition to the local pipeline. The attacker can suppress the local pipeline but not the external copy.

3. **Syncthing conflict detection:** Monitor for unexpected file changes synced from broker-server. Add a pre-sync hook on workstation that alerts when source code files change without a corresponding git commit.

4. **Separate the security pipeline from the monitored infrastructure.** At minimum, move Wazuh Manager to edge-node or a cloud VM. The machine that detects attacks should not be the machine most likely to be attacked.

### Should-Do

5. **NATS clustering:** Run NATS on 3 machines (broker-server, workstation, edge-node) in a cluster. Compromise of one node doesn't give control of the stream. This also provides availability -- if broker-server goes down, NATS stays up.

6. **Broker read replicas:** Run a read-only broker replica on workstation that independently verifies fleet state. If the primary broker reports different data than what NATS events would produce, something is wrong.

7. **Hardware security keys for SSH:** Use FIDO2/U2K security keys for SSH auth to broker-server. Even if an attacker has the machine, they can't use the SSH keys without the physical hardware key.

8. **Immutable infrastructure:** Run services in containers with read-only root filesystems. The attacker can compromise the running process but can't persist changes to the filesystem.

## Architectural Weakness

broker-server is the brain, the eyes, the mouth, and the immune system of the Sontara Lattice. If an attacker takes it, they control:
- What the fleet sees (Wazuh, NATS events)
- What the fleet remembers (fleet memory, broker database)
- What the fleet does (daemons, response actions)
- How the fleet communicates (broker, peer messaging)
- Whether the fleet knows it's been attacked (security-watch, response-daemon, email alerts)

There is no independent verification path. The monitoring systems that should detect the compromise are running on the compromised machine. Gridwatch on edge-node polls the compromised broker. iot-device's sentinel publishes to the compromised NATS. The daemons that should investigate are running on the compromised host.

The only reliable detection is out-of-band: Tailscale admin panel, GCP billing, GitHub audit logs, or physical access to another machine. None of these are currently integrated into the automated detection pipeline.

This is not a flaw that can be patched. It's an architectural decision that needs to be reversed. Critical services must be distributed across multiple machines so that no single compromise is total.
